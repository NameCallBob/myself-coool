# Case Study 內容蒐集任務 — 給專案 AI Agent 的完整指令

> **使用方式**:把這整份文件貼給目標專案 repo 裡的 AI agent(Claude Code / 其他),
> agent 讀完後應在該 repo 內執行蒐集與驗證,最後輸出一份符合本規格的 Markdown 報告。
> 報告會由 binbin 改寫成 [binbinbob.work](https://binbinbob.work) 的雙語 case study,
> 對應 `content/projects.ts` 的固定 schema。

---

## 一、你的任務(給 agent)

你在一個軟體專案的 repo 裡。你的任務是**為這個專案產出一份作品集 case study 的原始素材報告**,內容將公開發布在 binbin 的個人作品集網站上,讀者是資深工程師、技術主管與面試官。

你要做三件事:

1. **命名**:為專案提出一個專業、可公開的名稱(中英文各一)。
2. **蒐集**:用可重跑的指令從 repo、git 歷史、設定檔中蒐集事實。
3. **輸出**:依本文件第五節的 12 個章節,輸出一份 Markdown 報告。

**不要**直接寫行銷文案。你的產出是「經過驗證的事實 + 佐證指令」,文案由 binbin 統一改寫。

---

## 二、最高原則(違反任何一條,整份報告作廢)

1. **每個數字都要能重跑驗證**。任何數量(模組數、模型數、端點數、行數、commit 數、測試數)
   都必須附上得出該數字的指令與執行結果。跑不出來的數字不要寫。
2. **不誇大、不編造、不估計**。沒有的數據直接寫「無」。
   「大約」「超過」只能用在真的無法精確計數的情況,且要說明原因。
3. **狀態如實**:未上線就寫未上線;驗收中就寫驗收中;修到一半就寫修到一半
   (例:「35 項發現,30 項已修復、5 項部分修復,待 staging 複驗」)。
   誠實記錄的中間狀態比假裝完成更有說服力。
4. **區分「我做的」與「別人做的」**。多人專案必須用 git 身分切分貢獻,
   不可把團隊成果寫成個人成果。接手專案要明確標出接手時間點與接手前後的分界。
5. **禁止行銷語言**:不用最高級形容詞(最強、極致、完美)、不用「大幅提升」這種
   無基準的說法。有 before/after 數字就給數字(如「1 FPS → 即時」「5,256 行 → 230 行」)。
6. **遵守揭露邊界**(見第六節)。有疑慮的內容標記 `⚠️ 待確認可否公開`,不要自行判斷放行。

---

## 三、Step 0 — 專案命名

內部系統不能用公司內部代號或品牌名。請提出 **2–3 組候選名稱**,每組含:

- **中文名**:描述系統本質的專業名稱,格式參考:
  - 「多租戶 SaaS 人資系統」(形態 + 領域)
  - 「高科大設備借用管理系統」(機構 + 領域,僅限可公開機構)
  - 「大型 Django 微服務平台」(規模 + 技術 + 形態)
- **英文名**:對應的專業英譯,如 `Multi-tenant SaaS HRIS`、`NKUST Equipment Borrowing System`
- **slug**:kebab-case、全小寫,如 `hris-saas`、`equipment-borrowing`
- **命名理由**:一句話

命名規則:
- 用「架構形態 × 業務領域」描述,不用內部代號、不用客戶/公司名(除非確認可公開)
- 讓資深工程師看到名稱就能猜到系統的規模與複雜度
- 避免 AI 味的空泛詞(Smart、Next-gen、Revolutionary)

## 四、Step 1 — 事實蒐集程序

### 4.1 先確認 git 身分

執行 `git shortlog -sne --all`,列出所有作者。
**把結果列給 binbin 確認哪些身分屬於他**(他有多組 git 身分,如 `NameCallBob`),
再計算個人貢獻。多人專案必算:

```bash
# 各作者 commit 數
git shortlog -sne --all
# binbin 的 commit 數(確認身分後代入)
git log --all --author="<email-or-name>" --oneline | wc -l
# 時間範圍
git log --all --author="<...>" --format="%ad" --date=format:"%Y-%m" | sort | sed -n '1p;$p'
# 接手型專案:接手日前後分界
git log --all --before="YYYY-MM-DD" --oneline | wc -l
git log --all --since="YYYY-MM-DD" --author="<...>" --oneline | wc -l
```

### 4.2 規模數字(依技術棧選用,全部附指令與輸出)

**Django / DRF 後端:**
```bash
# app 數(排除第三方)
ls <apps-dir> | wc -l   # 或從 INSTALLED_APPS 數本地 app
# 資料模型數
grep -rE "class \w+\(.*models\.Model" --include="*.py" | wc -l
# 資源路由數
grep -rE "router\.register" --include="*.py" | wc -l
# 自訂 action 數
grep -rE "@action\(" --include="*.py" | wc -l
# 測試函式數
grep -rE "def test_" --include="*.py" | wc -l
```

**React / 前端 SPA:**
```bash
# 頁面/路由數(依 router 設定數)
# service 層檔案數與 API 呼叫點
ls src/services | wc -l
grep -rE "(axios|request|fetch)\.(get|post|put|patch|delete)" src/ | wc -l
# 行數
find src -name "*.tsx" -o -name "*.ts" -o -name "*.jsx" -o -name "*.js" | xargs wc -l | tail -1
```

**Flutter / 行動端:**
```bash
find lib -name "*.dart" | wc -l
find lib -name "*.dart" | xargs wc -l | tail -1
ls lib/screens | wc -l   # 依實際目錄結構調整
grep -rE "class \w+ extends (StatelessWidget|StatefulWidget|ConsumerWidget)" lib/ | wc -l
```

**通用:**
```bash
# CI pipeline 數與內容
ls .github/workflows/
# 測試檔數(單元 / 整合 / E2E 分開數)
# 第三方整合:從依賴檔(requirements/package.json/pubspec.yaml)列出
#   金流、推播、登入、監控等關鍵整合
```

### 4.3 質性素材來源(按可信度排序)

1. **git log**:重大重構、修復、功能的 commit 序列(git 是最可信的證據;
   CHANGELOG 若是事後補寫、日期對不上 commit,以 git 為準並註記)
2. **repo 內文件**:README、架構文件、稽核報告、遷移報表、測試報告
3. **設定檔**:CI、部署腳本、docker-compose、nginx conf(推斷部署形態)
4. **程式碼本身**:middleware、攔截器、基底類別等橫切設計

## 五、Step 2 — 輸出報告的 12 個章節

以下每章都標了對應的網站 schema 欄位與字數期望。**全部用中文條列**,雙語改寫由 binbin 處理。

### §1 專案命名(→ `title` / `slug`)
第三節的 2–3 組候選名稱與理由。

### §2 一句話定位(→ `oneLiner`,1–2 句)
系統是什麼、給誰用、目前狀態。狀態詞要精準:
`營運中` / `已部署` / `驗收修復階段` / `上架準備` / `開發中` / `接手維護`。
> 範例:「涵蓋出勤、排班、薪資、簽核到招募的多租戶人資平台,36 個功能模組;目前處於驗收與修復階段。」

### §3 範疇標籤(→ `scope` / `stack`)
- `scope`:一行短語,格式如「公司內部 · 驗收修復階段」「營運中 · 全端 + 維運」
- `stack`:4–6 個大寫技術標籤,如 `DJANGO`、`DRF`、`MYSQL`、`REDIS`、`REACT`、`FLUTTER`、`ONNX`

### §4 關鍵指標(→ `keyMetric`,1 個)
最能代表這個專案規模或品質的**單一數字**,附標籤與佐證指令。
> 範例:`728` 端點實測、`1,112` 後端測試函式、`82` 畫面

### §5 問題背景(→ `problem`,1 段、3–5 句)
系統出現之前的實際痛點 → 這個系統用什麼形態解決。寫給不熟悉該領域的工程師看得懂。
> 範例開頭:「中小企業的 HR 事務常散落在 Excel、紙本簽核與各自為政的工具之間⋯」

### §6 限制條件(→ `constraints`,2–4 條)
技術限制、資源限制(主機/預算/人力)、法規資安要求、時程壓力。
**第一條固定寫公開性**:「公司內部專案,程式碼不公開」或「已上線於 <可公開網域>」。
> 範例:「多租戶隔離是硬性要求:任何一條查詢漏掉租戶條件都是資料外洩。」

### §7 架構事實(→ `architecture`,2 段)
- **第 1 段:整體形態與規模**——單體/模組化單體/多服務、框架與版本、
  模組數/模型數/路由數/端點數(全部來自 4.2 的實測),以及**為什麼選這個形態**的一句話。
- **第 2 段:橫切設計與基礎設施**——多租戶/權限/快取/背景任務/即時通訊機制、
  資料庫、第三方整合、CI/CD、監控。
> 語氣範例:「刻意不拆微服務——薪資與簽核這類跨模組流程在單一交易邊界裡簡單得多。」

### §8 我的角色(→ `responsibilities`,1 段)
- 參與程度誠實分級:`獨立開發` / `主導` / `主要開發` / `參與` / `接手維護`
- 起訖年月、commit 數與佔比(4.1 的實測)
- 多人專案:**負責與不負責的部分各是什麼**;接手專案:接手前既有 vs 接手後我做的
> 範例:「2025 年 11 月接手;此前的基礎架構與商城流程由原團隊建置。接手後⋯(該期間 121 個 commit、占 85%)」

### §9 挑戰 → 解法(→ `challenges`,3 組;**整份報告最重要的部分**)
每組包含:
- **挑戰 c**:具體問題,有場景、有症狀、有數字。
  ❌「效能不好」 ✅「接手時的 AR 試戴只有 1 FPS,偵測時設計圖還會閃爍漂移」
- **解法 s**:做了什麼(考慮過哪些方案、為何選最後的)、結果如何。
  結果優先給 before/after 數字;沒有數字就描述可觀察的行為改變。
  修復未完成就寫實際狀態。

好的挑戰來源(優先挑這些):資安稽核發現與修復、效能瓶頸重寫、
髒資料遷移、大檔重構、認證/快取/併發的隱蔽 bug、無預算下的自建方案。

### §10 量化事實(→ `facts`,4 個)
4 個「數字 + 短標籤」,會以統計磚呈現。從 4.2 實測中挑最有代表性的,
**類型盡量分散**(規模 1–2 個、品質/測試 1–2 個、貢獻 0–1 個)。

### §11 學到什麼(→ `lessons`,2 條)
每條 = 一個誠實的反思:根因層級的教訓、會改做什麼、承認的技術債。
這部分越誠實越有說服力,禁止寫成自誇。
> 範例:「快取設計就是權限設計——薪資快取洩漏的根因,是快取鍵少了身分維度。」
> 範例:「重來會直接選有原生 SSR/SSG 的框架,省掉整條自維護管線。」

### §12 揭露邊界與佐證附錄
- **揭露邊界**:明確列出這個專案**不可公開**的項目(見第六節),
  以及報告中每一處 `⚠️ 待確認可否公開` 的清單。
- **佐證附錄**:一張表,`數字 | 指令 | 執行輸出摘要 | 出處檔案`,涵蓋報告中所有數字。

## 六、揭露邊界(公司內部專案必守)

**絕對不可出現在報告正文**(可另列在「僅供 binbin 內部參考」區塊):
- 公司名、品牌名、客戶名、內部專案代號(除非 binbin 明示可公開)
- 任何網域、IP、內網位址、伺服器主機名
- 內部人員姓名、Email、git email 明碼(統計貢獻時用「兩組 git 身分」這類描述)
- 營收、金額、客戶數等商業數據
- **現存未修復弱點的可重現細節**(已修復的漏洞可以寫,且是好素材;
  寫法聚焦根因與修法,不給攻擊路徑)
- API key、secret、憑證(蒐集過程若發現,單獨警示 binbin,不寫進報告)

**可以公開**:技術棧、架構形態、規模數字、已修復漏洞的類型與修法、
測試與 CI 實務、我的貢獻範圍與 commit 統計。

## 八、品質自檢(輸出前 agent 自問)

- [ ] 報告中每個數字都在佐證表裡,且指令可重跑?
- [ ] 有沒有任何一句話是「聽起來很棒但無法驗證」的?刪掉或降級成可驗證版本。
- [ ] 多人專案的貢獻切分清楚了嗎?會不會被誤讀成整個系統都是一人做的?
- [ ] 專案目前狀態(上線/驗收/開發中)寫得夠精準嗎?
- [ ] 揭露邊界掃過一遍了嗎?公司名/網域/人名/金額/未修復弱點細節?
- [ ] 3 組挑戰是否都「有場景、有症狀、有結果」,而不是抽象的技術名詞堆疊?
