# 作品集內容需求規格(給各專案 AI Agent)

> 用途:binbinbob.work 個人網站的 case study 內容來源。
> 目標語氣:**可信地陳述經驗**——只寫做過、可驗證的事;不用行銷語言、不用最高級形容詞;
> 規模数字要能說出「怎麼算的」。寧可保守,不可誇大。
> 請用中文條列回答,我會負責改寫成網站文案與英文版。

## 對每個專案,請提供以下 10 項

### 1. 一句話定位
這個系統是什麼、給誰用、目前狀態(上線中/內部使用/開發中)。

### 2. 我(binbin)的角色與時間範圍
- 參與程度誠實描述:獨立開發 / 主導 / 參與 / 接手維護
- 起訖時間(年月)
- 若是多人專案:我負責與**不負責**的部分各是什麼

### 3. 問題背景(2–4 句)
這個系統出現之前,使用者/組織的實際痛點是什麼?為什麼需要它?

### 4. 限制條件(2–4 條)
技術限制、資源限制(主機/預算/人力)、時程、法規或資安要求。

### 5. 架構事實(只要可由 repo 驗證的)
- 整體形態:單體 / 模組化單體 / 多服務,前後端框架與版本
- 模組/app 數、資料模型數、API endpoint 數(附計算方式,如 grep 指令)
- 資料庫、快取、背景任務、即時通訊、第三方整合(金流/推播/登入)
- 部署方式(主機、web server、CI/CD 有無)

### 6. 挑戰 → 解法(2–3 組,這是最重要的部分)
每組請寫:
- 遇到的**具體**問題(有場景、有症狀,不要抽象)
- 當時考慮過哪些做法、為什麼選了最後的做法
- 結果如何(有數字最好;沒有就描述行為改變)

### 7. 可公開的量化成果
效能數字、測試數量與種類、上線時間、實際使用者/單位——
**沒有的項目直接寫「無」**,絕對不要估計或編造。

### 8. 學到什麼 / 會改做什麼(1–2 條)
誠實的不足、技術債、下一步計畫。這部分越誠實越有說服力。

### 9. 揭露邊界
哪些不可公開:公司/客戶名、網域、內部人員、金額、現存弱點。
請明確列出,我會在文案層過濾。

### 10. 佐證方式
上述每個數字的出處(指令、檔案位置、報告),方便抽驗。
